# fwqgo 设计文档

更新日期：2026-07-15

本文档记录 `fwqgo` 的公开前台与 CMS 后台设计规则。公开前台是用户访问和 SEO 收录的主站，CMS 后台是内容生产和运营系统。前台设计信息放在最前面；后台规范保留在后半部分，避免前后台规则混用。

## A. 公开前台 Design 信息

适用范围：

- `apps/web`
- `src/features/public`
- 前台实际复用的公共 UI 组件
- 前台相关样式 token：`src/styles/globals.css`

不适用范围：

- `apps/cms`
- `src/features/cms`
- CMS 后台菜单、任务中心、表单和管理表格

### A1. 前台产品定位

前台不是普通博客，也不是营销落地页。它应被设计成：

```text
服务器优惠信息站 + 服务器套餐比价工具 + 技术内容媒体 + SEO 聚合入口
```

用户的核心任务：

1. 快速找到可买套餐。
2. 比较价格、地区、线路、库存状态和优惠码。
3. 打开购买链接。
4. 阅读推广文章或测评文章确认细节。
5. 通过分类、标签、地区、线路和英文页面继续浏览。

设计重心：

- 首页不应被大文章封面主导，首屏必须露出服务器比价、搜索、快速筛选和高价值专题入口。
- 完整套餐筛选、排序和对比功能集中放在 `/servers` 工具页，首页只展示少量精选套餐或热门入口。
- 信息密度优先于视觉装饰。
- 可扫描性优先于叙事感。
- SEO 内容和购买转化并重。

### A2. 前台信息架构

公开前台主要页面：

```text
/                              中文首页
/en                            英文首页
/servers                       服务器比价工具页
/servers/hong-kong             香港服务器专题
/servers/united-states         美国服务器专题
/servers/cheap-vps             便宜 VPS 专题
/servers/providers/[provider]  商家套餐集合页
/servers/regions/[region]      地区套餐集合页
/servers/lines/[line]          线路套餐集合页
/fwq/[category]/page/[pageNo]  中文分类文章页
/fwq/tags/[tagSlug]/page/[pageNo] 中文标签页
/fwq/posts/[slug]              中文文章详情页
/en/fwq/...                    英文分类、标签、文章详情
/search                        搜索页
/go/[token]                    短链跳转
```

主导航优先级：

1. 服务器比价
2. 香港服务器
3. 美国服务器
4. 便宜 VPS
5. 文章分类
6. 搜索
7. 中英文切换

导航规则：

- 主导航不能只展示文章分类，必须突出服务器比价和三类高价值专题。
- 桌面端允许下拉分类；移动端使用 sheet 菜单。
- 文章分类是辅助浏览入口，服务器套餐入口是首要转化入口。
- 英文站导航保留同样结构，只改变文案和路径。

### A3. 前台 Design Tokens

当前前台基础 token 来自 `src/styles/globals.css`。

颜色：

```text
background: 0 0% 100%
foreground: 222.2 84% 4.9%
primary: 221.2 83.2% 53.3%
accent: 221.2 83.2% 53.3%
muted: 210 40% 96.1%
muted-foreground: 215.4 16.3% 46.9%
border: 214.3 31.8% 91.4%
destructive: 0 84.2% 60.2%
```

语义使用：

- `primary`：主操作、主链接、当前状态、重要数字。
- `accent`：前台品牌强调色，目前与 `primary` 同色。
- `muted`：弱背景、筛选容器、空态背景。
- `border`：卡片、表格、输入框边界。
- 绿色：有货、成功、可购买。
- 琥珀色：预售、补货、需确认。
- 灰色：缺数据、未核验、停售。
- 红色：错误、危险、不可恢复操作。

字体：

```text
font-ui: Inter, SF Pro Text, Segoe UI, PingFang SC, Microsoft YaHei, Noto Sans CJK SC, Helvetica Neue, Arial, sans-serif
font-editorial: 同 font-ui
```

字号建议：

```text
页面 H1: 28-40px，首页不超过 48px
页面 H2: 22-28px
卡片标题: 16-18px
正文: 16-17px
辅助说明: 13-14px
表格/标签: 12-14px
```

圆角：

```text
按钮、输入框、筛选器: rounded-md
普通卡片: rounded-lg
大型专题容器: rounded-lg
避免大面积 rounded-2xl
```

间距：

```text
页面 section 垂直间距: 32-48px
首屏内部 gap: 16-24px
卡片 padding: 12-20px
表格 cell padding: 8-12px
筛选控件高度: >=44px
移动端可点击区域: >=44px
```

### A4. 前台视觉方向

关键词：

```text
专业、可信、清晰、紧凑、技术媒体、价格比较、可购买
```

推荐视觉：

- 浅色背景 + 细边框 + 克制阴影。
- 使用真实封面图，但不让图片支配首屏。
- 表格、筛选器、列表是前台核心视觉语言。
- 文章页正文保持高可读性，链接明显可点击。
- 状态、价格、优惠码使用清晰标签和数字层级。

禁止或尽量避免：

- 纯营销 hero。
- 大面积留白。
- 装饰性渐变光斑。
- 玻璃拟态叠加过多。
- 单色调蓝色页面。
- 图片暗遮罩导致标题看不清。
- 卡片套卡片。
- 只靠颜色表达状态。

### A5. 首页设计规格

首页目标：

```text
让用户在首屏理解网站价值，并快速进入 “搜索、服务器比价、专题页、最新优惠文章”。
首页负责入口分发和少量精选推荐，不承载完整套餐筛选工具。
```

推荐桌面结构：

```text
Header
└─ Sticky 主导航

Hero / Entry Console
├─ 左侧：标题 + 简短说明 + 搜索框 + 快速筛选 chips
├─ 中间：服务器比价入口 + 三大高价值专题入口
└─ 右侧：最新优惠码 / 热门商家 / 热门地区 / 精选套餐

服务器比价入口
├─ 全部服务器比价
├─ 香港服务器
├─ 美国服务器
└─ 便宜 VPS

最新优惠与评测
├─ ArticleCard 列表
└─ 右侧信息栏：站长推荐 / 高浏览量 / 最新套餐

Footer
```

推荐移动结构：

```text
Header
搜索框
快速筛选横向 chips
服务器比价入口
服务器专题入口
精选套餐卡片 3-5 条
最新文章卡片
底部导航/页脚
```

首页设计规则：

- H1 控制在两行以内。
- 首屏不放超大文章封面。
- 首页文章大图只能作为次级模块，不能压过服务器比价入口和搜索入口。
- 快速筛选必须包含高价值意图：香港 CN2、香港 CMI、美国 VPS、便宜 VPS、独立服务器、优惠码。
- 右侧栏必须是真实模块，不使用“占位内容位”。
- 首页可以展示少量精选套餐，至少包含价格、地区、线路、商家、状态；完整筛选、排序、对比放在 `/servers`。
- 首页不渲染完整大表格，避免变成后台式工具页。

首页推广位：

- 首页推广文章、精选套餐和推广图片统一存入 `homepage_slots`。
- 支持 `hero_primary`、`promo_grid`、`featured_offers`、`sidebar` 四个固定位置。
- 每个推广位包含语言、排序、启用状态、上线时间和下线时间；前台只读取当前有效配置。
- 新推广位尚未配置或读取失败时，继续使用旧首页推荐文章作为兼容兜底。
- 推广图片保持稳定 `16:9` 比例，图片本身不承担完整说明，标题和目标链接必须独立可读。

### A6. Header 设计规格

当前实现：`src/features/public/components/header.tsx`

桌面端：

- 高度：约 64px。
- 左侧品牌。
- 中间导航优先展示服务器比价和分类。
- 右侧语言切换。
- 导航菜单使用 `NavigationMenu`，下拉宽度控制在 420-620px。

移动端：

- 只保留品牌和菜单按钮。
- 菜单使用右侧 sheet。
- Sheet 内先展示套餐专题，再展示文章分类。
- 所有移动菜单项高度不小于 44px。

Header 不应：

- 只把品牌作为首页入口而隐藏核心导航。
- 在移动端展示过多横向 nav。
- 把文章分类放在服务器比价前面。

### A7. ArticleCard 设计规格

当前实现：`src/features/public/components/article-card.tsx`

用途：

- 首页最新文章。
- 分类页。
- 标签页。
- 搜索结果文章区。
- 英文文章列表。

结构：

```text
ArticleCard
├─ 16:9 封面
├─ 主标签 + 发布时间
├─ 标题
├─ 摘要
├─ 次级标签
└─ 阅读入口
```

规格：

- 封面固定 16:9。
- 桌面端图片宽度约 220-240px。
- 移动端图片在上，内容在下。
- 标题 2 行截断。
- 摘要 2 行截断。
- 标签最多显示 4 个，主标签优先。
- 阅读按钮文字可见，不只显示箭头。
- hover 时标题必须下划线或明显变色。

图片 fallback：

- 图片不可用时显示稳定渐变/背景和文章标题。
- 不能出现破图。
- fallback 仍占据 16:9 空间，避免 CLS。

### A8. ServerOfferTable 设计规格

当前实现：`src/features/public/components/server-offer-table.tsx`

桌面端表格字段：

```text
套餐/商家
价格/周期
地区/线路
配置
状态
优惠码
购买
推广文章
测评文章
数据核验时间
```

筛选器：

- 关键词
- 商家
- 状态
- 地区
- 线路
- 排序
- 是否有优惠码

排序：

- 默认价格从低到高。
- 允许价格从高到低。
- 允许最新优先。
- 缺失价格和非法价格排到最后。

移动端：

- 使用套餐卡片，不强制横向拖表。
- 卡片首屏必须包含标题、商家、价格、地区/线路、状态、购买按钮。
- 配置和有效期可放在下方。

链接安全：

- 购买链接：`nofollow sponsored noopener noreferrer`。
- 外链：新窗口打开。
- 站内文章链接：Next `Link`。
- 非 `http/https` 或非站内路径不渲染为可点击入口。

### A9. 服务器比价工具页与专题页设计规格

适用：

- `/servers`
- `/servers/hong-kong`
- `/servers/united-states`
- `/servers/cheap-vps`
- `/servers/providers/[provider]`
- `/servers/regions/[region]`
- `/servers/lines/[line]`

页面结构：

```text
工具/专题页头
├─ 返回服务器比价（专题页）
├─ Badge
├─ H1
├─ 简短说明
└─ 数据摘要

套餐区
├─ 筛选器
├─ 当前匹配数量
├─ 桌面表格 / 移动卡片
└─ 空态

SEO 内容
├─ 常见问题
└─ 相关专题链接
```

规则：

- `/servers` 是完整比价工具页，应直接提供全量或主要套餐的筛选、排序和对比，不只展示专题卡片。
- `/servers/hong-kong`、`/servers/united-states`、`/servers/cheap-vps` 是预筛选 SEO 专题页，承接具体搜索意图。
- 专题说明不要超过 2-3 行。
- 空 slug、非法 slug 返回 404 或空 metadata，不渲染空标题。
- 无套餐时显示明确空态，不渲染空表格。
- FAQ 用于 SEO，但不能压在套餐列表前面。

`/servers` 当前运行架构：

```text
静态页面壳（Header、标题、说明、专题入口、Footer）
└─ Suspense
   └─ 运行时库存区
      ├─ URL 筛选解析
      ├─ 数据库筛选和搜索
      ├─ 固定 30 条游标分页
      └─ 桌面表格 / 移动卡片
```

- 页面使用 PPR，筛选查询不会阻塞静态页面壳输出。
- 筛选状态全部进入 URL，支持关键词、厂商、产品组、库存、探测状态、地区、线路、特征、优惠码、价格区间和排序。
- 桌面端使用厂商侧栏和数据表格；移动端隐藏侧栏，使用筛选器和紧凑套餐卡片。
- 价格统一保存 `monthlyPriceUsd` 用于跨周期比较；原始价格和多个付款周期仍单独展示。
- 套餐没有价格但存在有效购买入口时，可以进入库存工具；价格为空时排在有效价格之后。
- 库存探测超过 24 小时由数据库查询计算为过期状态，React 渲染阶段不读取当前时间。

集合页和 SEO：

- 商家、地区、线路页使用规范化 slug；动态集合查询缓存 15 分钟，并由套餐变更事件主动失效。
- 有效可购买套餐少于 5 个的集合页输出 `noindex,follow`，同时不进入服务器 sitemap。
- `/servers` 带筛选参数的 URL 统一 `noindex,follow`，canonical 固定为 `/servers`，避免筛选组合产生重复收录。
- 服务器专题和集合页只展示最多 30 条精选套餐；大量数据检索统一引导回 `/servers` 工具页。

### A10. 文章详情页设计规格

适用：

- `/fwq/posts/[slug]`
- `/en/fwq/posts/[slug]`

顶部：

- H1 不超过桌面 40px。
- 发布时间、浏览量、复制链接、分享到 X 放在同一信息区。
- 摘要最多两行。
- 封面图桌面端不超过 360px 高，移动端保持 16:9。

正文：

- 正文字号 16-17px。
- 行高 1.7-1.8。
- 链接必须下划线。
- 链接 hover 颜色明显变化。
- 表格尽量一屏可读，字体和 padding 可以适当缩小。
- `h2` 使用左侧强调线或底部分隔。
- `h3` 使用浅背景或边框提升识别度。

目录：

- 桌面端显示。
- 1280-1535px 优先保留正文与右侧栏，目录从 1536px 的 `2xl` 断点开始显示，避免三栏挤压正文。
- 小屏隐藏。
- 随滚动高亮当前章节。
- 最大高度可滚动，不能遮挡正文。

底部：

- 相关套餐优先于泛推荐文章。
- 标签、上一篇/下一篇放在正文后。
- 相关推荐包含专题页、相关套餐、相关文章。

文章套餐分层：

- 从当前文章直接提取的套餐可以在正文前使用紧凑速览展示。
- 仅通过标签、地区或线路推测匹配的套餐放在正文后，不能伪装成本文明确提及的套餐。
- 封面保持完整显示，桌面端最大展示区域为 `640 × 360`，不通过裁切隐藏图片内容。

### A10.1 标签页设计规格

- 标签页头使用扁平信息带，不使用统计卡片嵌套卡片。
- 页头只保留标签名、描述、文章总数和当前页，桌面高度控制在约 220px 内。
- 文章卡片不重复展示当前标签，优先显示其他关联标签。
- 相关套餐只在关键词、商家、地区或线路有有效命中时展示。
- 右侧栏展示当前主题命中的商家、地区、线路和服务器比价入口，不使用泛化的“标签说明”。
- `indexable=false` 的标签页输出 `noindex,follow`；不存在的标签和越界分页返回 404。
- 中文与英文标签页保持相同结构，中文页面补齐 `px-4 sm:px-6` 移动端留白。

### A11. 搜索页设计规格

适用：`/search`

搜索能力：

- 文章标题。
- 文章描述。
- 标签。
- 商家。
- 地区。
- 线路。
- 优惠码。
- 套餐配置。

结果展示：

```text
搜索框
匹配摘要
套餐结果
文章结果
相关标签/专题建议
空态建议
```

规则：

- 套餐结果和文章结果分组，不混成一个列表。
- 空态给出可点击建议：香港服务器、美国 VPS、便宜 VPS、优惠码。
- 搜索参数进入 URL，便于分享和返回。

### A12. 图片和封面设计

通用：

- 文章封面统一 16:9。
- 首页、列表、专题页所有图片必须有稳定尺寸。
- 非首屏图片懒加载。
- 图片失败显示 fallback，不显示破图。

AI 生成封面方向：

- 用真实可理解的服务器、网络线路、价格对比、地区主题。
- 不做抽象光效图。
- 中文文章封面命名带 `zh`。
- 英文文章封面命名带 `en`。

封面使用场景：

```text
文章详情页：大图，但高度受控。
首页文章卡片：小图辅助识别。
分类/标签页：列表缩略图。
搜索页：结果缩略图。
```

### A13. 前台 SEO 和性能

SEO：

- 中文主版本使用根路径。
- 英文版本使用 `/en/`。
- 所有文章、分类、标签、服务器专题必须有 canonical。
- 有中英文对应时输出 `alternates.languages`。
- 标签页只让有搜索价值的标签进入 sitemap。
- 套餐专题承接高转化关键词。

性能：

- 前台静态优先，优先 ISR/PPR。
- 避免公开页面大量 `no-store`。
- 非必要不强制动态渲染。
- 首页和专题页图片必须优化。
- 表格和卡片避免布局跳动。
- 搜索和筛选状态应避免阻塞首屏。
- Web 运行时只使用只读数据库连接；浏览量等埋点使用独立写连接。
- 首页、专题、集合和筛选 facets 使用独立缓存生命周期，不共享一个全站粗粒度缓存。
- CMS 变更通过受保护的 `/api/internal/revalidate` 通知 Web，失败时重试三次，最终仍由缓存时间策略兜底。

### A14. 前台移动端验收

375px 宽度必须满足：

- 页面无横向滚动。
- Header 不挤压品牌。
- 菜单按钮触控区不小于 44px。
- 快速筛选 chips 可换行或横向滚动，但不撑破页面。
- 文章卡片封面保持 16:9。
- 套餐卡片不依赖横向拖动阅读核心信息。
- 购买按钮、文章按钮、标签按钮都可轻松点击。
- 正文表格不遮挡页面，必要时内部横向滚动。

### A15. 前台落地检查清单

改前台 UI 前确认：

- 是否只影响 `apps/web`、`src/features/public` 或确实被前台使用的共享组件。
- 是否没有把 CMS 后台工作台风格带到公开前台。
- 是否保留服务器比价入口的高优先级。
- 是否让购买入口和文章入口都清楚可见。
- 是否保留 SEO 结构和中英文路径。
- 是否所有外链安全属性正确。
- 是否所有图片都有固定比例或明确尺寸。
- 是否移动端无横向溢出。

建议验证命令：

```bash
npm run typecheck
./node_modules/.bin/eslint .
SKIP_ENV_VALIDATION=1 npm run build
```

## B. CMS 后台设计信息（保留）

## 1. 后台定位

CMS 后台不是普通文章后台，而是服务器/VPS 推广内容的生产系统。

核心职责：

- 抓取、清洗和保存素材。
- 改写中文文章，翻译英文文章。
- 单独生成中英文 SEO 字段。
- 提取服务器套餐并人工校正。
- 管理返利商家、短链、首页推荐和外链替换审计。
- 管理上传图片、AI 生图、文章封面图。
- 可视化长任务的步骤、产物、token、模型、失败原因和重试入口。
- 审核草稿并发布到公开前台。

后台始终中文优先。英文只作为内容数据和前台输出语言存在，后台导航、按钮、toast、错误提示和操作说明都应使用中文。

## 2. 应用边界

项目采用前后台分离的 App Router 结构：

- `apps/web`：公开前台，默认端口 `3000`。
- `apps/cms`：CMS 后台，默认端口 `3100`。
- `src/features/public`：公开前台路由和组件。
- `src/features/cms`：后台路由、组件、数据读取和 Server Actions。
- `src/features/shared`：前后台共享实现。
- `packages/db`：Drizzle schema、数据库 client 和数据 helper。
- `packages/ai`：AI 改写、翻译、任务 runner、源站拉取。
- `packages/scrape`：抓取、清洗、HTML/Markdown 转换、返利链接替换。

后台变更不要无意影响公开前台。涉及公开页面缓存的后台发布动作，需要调用现有 cache revalidation 机制。

## 3. 后台信息架构

当前后台一级导航：

- 概览：后台运营总览入口。
- 内容：AI 生产台、文章库、AI 任务中心、发布质检。
- 套餐：套餐管理、库存监控。
- 媒体：图片资产、AI 生图。
- 运营：首页运营、链接管理、SEO 管理。
- 模型与接口：AI 改写和生图接口设置。

合并规则：

- 草稿与已发布文章统一进入“文章库”，通过状态和语言筛选；旧草稿地址重定向并保留筛选参数。
- 上传图片作为图片资产页的主操作，不单独占用侧栏入口。
- 通用生图和文章封面使用同一组页内 Tab。
- 返利商家和短链跳转合并为“链接管理”的页内 Tab。
- 主页、分类和标签 SEO 合并为“SEO 管理”的页内 Tab。
- AI 改写与生图接口配置使用同一组“模型与接口”页内 Tab。
- 旧功能路由继续保留，侧栏只展示高频工作入口。

设计原则：

- AI 生产台只负责发起和配置生产，不重复展示完整任务列表；任务列表集中在 AI 任务中心。
- 筛选、页码和语言条件应进入 URL，刷新、复制链接、返回页面后保持状态。
- 列表页、草稿箱、AI 任务、图片列表、返利商家和套餐列表都应遵守同一套 URL 状态规则。
- 中英文文章是两篇独立文章，但后台应显示对应关系，支持跳转、缺失提示和重新生成英文。

## 4. 认证与安全

后台访问保护：

- 生产域名使用 `cms.fwqgo.com`。
- 生产环境应启用 Basic Auth + 管理员登录双层保护。
- 登录使用 HTTP-only session cookie。
- 所有后台页面和后台 mutation 必须依赖服务端 session 校验。
- Admin Server Action 在读写受保护数据前必须调用 `requireAdminSession()` 或项目现有等价 helper。
- API 和 Server Action 不应向前端泄漏原始异常栈，应统一返回可读错误结构。

公开注册默认关闭，只能通过 `ENABLE_PUBLIC_SIGNUP=true` 显式开启。

## 5. UI 设计系统

后台 UI 风格应是数据密集、工作台式、安静实用的运营系统。

基础规则：

- 使用现有 shadcn/Radix 组件和 Tailwind token，不引入大型新 UI 框架。
- 使用 Lucide 图标，不用 emoji 作为结构性图标。
- 页面第一屏优先展示核心操作和关键数据，不做营销式 hero。
- 页面区块保持扁平和清晰，不嵌套卡片，不用装饰性渐变球或无意义大背景。
- 字体尺寸按场景控制，后台表格、筛选器和工具栏不使用夸张标题。
- 中文说明允许换行，不要把长操作说明 line-clamp 掉。

交互与无障碍规则：

- 主要按钮、图标按钮、选择器、输入框、分页、Toast 操作的可点击区域尽量不小于 `44px`。
- 图标按钮必须有 `aria-label` 或可感知文本。
- focus ring 必须清晰，不能移除键盘可见焦点。
- 禁用状态要有真实 `disabled` 或 `aria-disabled`，并有可见弱化。
- 加载超过短时间应使用 skeleton、按钮 loading 或任务状态反馈，不只让页面空白。
- Toast 文案必须说明结果、失败原因和下一步建议。

表格和响应式规则：

- 宽表格使用内部横向滚动和稳定 `min-width`，不要挤压到列内容互相覆盖。
- 移动端优先保留筛选、搜索、核心操作和状态信息，次要列可以折叠或横向滚动。
- 搜索清空按钮、复制、打开、删除、重试等小按钮也要保持足够触控区。
- 表格空态要说明原因，并在合适时提供下一步操作。

## 6. 页面 Shell 和通用组件

后台页面应优先复用：

- `AdminPageShell`：页面标题、描述、操作区、统计摘要。
- `AdminTableWorkbench`：搜索、筛选、批量操作、空态和表格容器。
- `AdminLoading`：后台统一 loading/skeleton。
- `PaginationComponent`：URL 友好的分页。
- `AppSidebar`、`AppBreadcrumb`：后台导航和当前位置。

约定：

- 页头描述可以是操作说明，必须可换行。
- 面包屑根路径显示“数据面板”，不能空白。
- 后台 sidebar 折叠/展开按钮和文案使用中文。
- 分页禁用态应可见，不要直接隐藏上一页/下一页导致布局跳动。
- select、toggle、pagination、breadcrumb、toast 等基础组件的交互尺寸和中文可访问文本应保持一致。

## 7. 内容生产主流程

文章生产链路：

```text
输入 URL / 文本 / 邮件
-> 抓取原始 HTML
-> 清洗正文、标题、图片和外链
-> 保存正文预览
-> 识别商户
-> 匹配返利链接
-> 改写中文文章
-> 生成中文 SEO 字段
-> 生成中文标签
-> 生成中文封面图
-> 从改写后的中文文章翻译英文文章
-> 单独生成英文 SEO 字段
-> 生成英文标签
-> 生成英文封面图
-> 中英文分别进入草稿箱
-> 人工审核发布
```

关键数据规则：

- `正文预览` 保存清洗后的原始正文，不保存改写后的正文。
- 英文文章生成必须以清洗后的原始正文和改写后的中文文章为可追溯输入，其中正文翻译以改写后的中文文章为主。
- 英文 SEO 字段单独生成，不直接复制中文 SEO。
- 中文和英文文章单独存在草稿箱和文章列表，筛选条件必须支持语言。
- 中文编辑页只修改中文文章；英文文章修改区域和流程应在英文文章编辑中处理。

## 8. AI 任务中心

AI 任务中心是所有长任务的事实来源。

必须展示：

- 任务进度。
- 当前步骤和步骤日志。
- 每一步状态：`pending`、`running`、`success`、`failed`、`skipped`、`manual_required`。
- 输入 token、输出 token、模型、Max Tokens、temperature。
- 截断、超时、生图失败、外链未命中等可读失败原因。
- 每一步产物：原文、清洗后正文、改写中文、翻译英文、SEO 字段、封面图。
- 重试、取消、恢复、从指定步骤继续。

长任务必须后台任务化：

- 抓取、清洗、返利替换、中文改写、英文翻译、SEO 生成、生图、套餐提取都不应依赖页面长时间等待。
- 页面操作只负责创建任务或触发任务推进。
- 失败错误应保存到任务日志，不只显示一次 toast。

Max Tokens 规则：

- AI 改写配置中的 Max Tokens 必须能同时限制中文生成和英文生成。
- 中文改写、英文翻译、SEO 生成应是独立调用，英文不能被上一个中文进程的状态影响。
- 如果模型返回 `finish_reason === "length"`，错误提示必须明确是模型输出被截断，并提示调大 Max Tokens 或缩短输入/提示词。

## 9. 返利链接替换设计

返利链接替换不依赖 AI 记忆，只依赖数据库中的返利商家配置。

普通参数模式：

```text
原文链接：https://clients.zgovps.com/?cmd=cart&action=add&affid=23&id=146
命中商家：zgovps
数据库参数：affid
数据库值：33
替换结果：https://clients.zgovps.com/?cmd=cart&action=add&affid=33&id=146
```

规则：

- 先从原始链接匹配商家，再读取该商家的返利参数和值。
- 一篇文章通常只对应一家商家，匹配到后本篇文章只查一次商家返利数据，并复用到其他链接。
- 普通参数模式只修改该参数值，路径、域名、其他 query、顺序尽量保持不变。
- `affParam === "href"` 时整条链接替换为数据库中的返利链接，这个行为不能改。
- 表格单元格里每个购买链接可能不同，HTML 转 Markdown 和返利替换都必须保留每个 `href`，不能只读文字，也不能把多个链接统一成同一个 URL。
- 后台应提供替换审计：替换前、替换后、命中商家、命中参数、是否 href 整条替换。
- 未命中的外链保留原链接，并加 `rel="nofollow noopener"`，同时把任务标为 `manual_required` 提示人工审核。

## 10. 图片与封面图设计

图片中心管理两类资产：

- 手动上传图片：商家 logo、活动截图、产品图、控制台截图、促销 banner。
- AI 生成图片：文章封面、首页推荐图、活动专题图、列表卡片图。

处理规则：

- 上传或生成后统一进入图片资产库。
- 图片需要转换/优化为 WebP。
- 图片列表支持按类型、状态、文章、商家等维度筛选。
- 图片资产需要中文 alt 和英文 alt。
- AI 生图必须后台运行，失败原因要可读。
- 自动化生成中英文文章时，应在合适步骤自动触发封面图生成，减少人工补图。

封面命名规则：

- 中文文章封面使用 `-zh` 区分，例如 `some-slug-zh-cover.webp`。
- 英文文章封面使用 `-en` 区分，例如 `some-slug-en-cover.webp`。
- 批量生成、自动化任务和手动生图必须使用同一套命名规则。

## 11. 服务器套餐设计

服务器套餐用于结构化展示 VPS/独服优惠，不直接等同于文章。

套餐核心字段：

- 商户关系、商户 slug、套餐名称、厂商产品 ID、产品组。
- CPU、内存、硬盘、流量、带宽、IPv4/IPv6、虚拟化类型。
- 地区关系、线路关系，以及兼容历史数据的地区/线路文本。
- 多条价格、币种、付款周期、原价、折后价、标准美元月价。
- 优惠码、活动开始/结束时间、库存状态。
- 购买链接、返利链接、来源 URL、来源类型、审核状态、可见状态、人工锁定字段。
- 最近探测状态、探测时间、状态变化时间和探测历史。

后台设计规则：

- 套餐列表应支持状态、审核、展示、语言/地区/商户等筛选。
- 批量操作应支持批量发布、转草稿/隐藏、更新 SEO、提取套餐等任务化入口。
- 购买链接必须校验为安全 URL，外链打开使用 `noopener noreferrer`，返利链接可使用 `nofollow sponsored`。
- 宽套餐表格使用横向滚动，避免压缩字段导致误读。
- 同一厂商的厂商产品 ID 必须唯一；自动同步通过该 ID 匹配现有套餐，不自动创建未知套餐。
- 人工锁定的标题、状态、价格和购买链接不能被库存监控覆盖。
- 库存监控只允许访问公网 HTTP/HTTPS JSON 接口，限制超时、响应体积、条目数量和字段映射。
- 带秘密请求头的监控请求禁止跟随重定向，避免把凭据发送到未知主机。
- 每次监控写入检测记录；失败时保留可读错误，并把已映射套餐标为探测失败。
- 监控失败只更新由当前接口地址成功映射过的套餐，不影响同厂商其他监控负责的产品。
- 套餐探测记录默认保留 30 天；任一监控运行后都会清理全局过期记录，已停用或已删除监控不会留下永久历史。
- 立即执行和定时执行共享同一后台任务队列，重试完成前不能并发排入下一次相同任务。

## 12. 双语内容和 SEO

前台多语言约定：

- 中文为默认根路径，例如 `/fwq/posts/[slug]`。
- 英文使用 `/en/` 前缀，例如 `/en/fwq/posts/[slug]`。
- 分类和标签英文路由优先使用 `enName`、`enSlug`，缺失时才 fallback 到中文。

后台管理规则：

- 文章、草稿、分类、标签、首页推荐、SEO 配置都要保留语言筛选或双语字段。
- 中文文章和英文文章单独存储、单独发布、单独编辑。
- 后台应显示中英文对应关系，但不能把两篇文章当成同一条记录直接混改。
- 站点 SEO、分类 SEO、标签 SEO 都需要中文和英文配置。
- 长 SEO 描述使用 textarea，不使用单行 input。

SEO 输出规则：

- 中英文页面 canonical 指向自身。
- 中英文页面通过 `hreflang` 互相关联。
- 英文 SEO 根据英文内容生成，不直接机器翻译中文 SEO。

## 13. 错误处理与反馈

后台错误统一包装：

- Server Action 和 API 返回统一错误结构。
- 前端 toast 显示可读原因和下一步建议。
- 长任务错误写入任务步骤日志。
- 截断、超时、鉴权失败、配置缺失、外链未命中、生图失败应有不同提示。

推荐错误结构：

```ts
type AdminActionResult<T> =
  | { ok: true; data: T; message?: string }
  | {
      ok: false;
      error: {
        code: string;
        message: string;
        suggestion?: string;
      };
    };
```

前端反馈规则：

- 成功操作用简短 toast。
- 失败操作说明失败原因和可执行建议。
- 批量任务创建成功后提示去 AI 任务中心查看进度。
- 危险操作需要确认，不只依赖 toast 撤销。

## 14. URL 状态和批量操作

筛选和分页进入 URL：

- 文章列表。
- 草稿箱。
- AI 任务。
- 图片列表。
- 返利商家。
- 套餐列表。
- 首页推广位。

约定：

- 搜索条件改变时回到第一页。
- 点击页码和下一页不能丢失当前筛选。
- URL query 是列表状态的事实来源，不要用 effect 反复同步并覆盖 `pageNo`。
- 批量操作不要阻塞页面；重任务创建后台任务。

批量操作应覆盖：

- 批量发布。
- 批量转草稿/隐藏。
- 批量生成封面。
- 批量生成英文。
- 批量更新 SEO。
- 批量提取套餐。

## 15. 工程约定

代码层规则：

- 后台 mutation 必须服务端鉴权。
- 优先使用 Drizzle 和现有 DB helper。
- 不新增 deploy-time side effect 到 `postinstall`。
- 复杂 HTML、URL、Markdown 处理优先使用结构化 parser/helper，不用脆弱字符串替换。
- 改动共享组件时要检查前后台共同影响。
- 改动后台页面后至少跑 `npm run typecheck` 和 `npm run lint`；涉及跨应用路由、构建产物或共享组件时跑 `SKIP_ENV_VALIDATION=1 npm run build`。

设计验收清单：

- 后台文案为中文。
- 图标按钮有可访问标签。
- 控件触控区不小于 44px。
- 宽表格不挤压、不造成页面级横向溢出。
- 筛选和页码刷新后保留。
- 长任务不让页面等待到超时。
